3 / 27 The big picture

Journey of a prompt

Step 1-2: Your prompt arrives and is split into tokens. Consider a 22,800-token agentic session.

Step 3: The engine checks if these tokens match a cached prefix.
Hit = skip to the new tokens. Miss = process everything from scratch.

Step 4-5: Prefill runs. The KV cache fills. The first output token appears. This is TTFT.

Step 5-6: Decode generates tokens one at a time, each reading the full KV cache. The engine tries to retain the cache for the next turn, but eviction pressure can force a re-prefill.

Walk through the diagram step by step. This is the slide where the audience builds their mental model of the full request lifecycle. Emphasize the branch at step 3: cache hit vs miss. This single decision point is where all the systems engineering matters. A cache hit means you skip potentially hundreds of milliseconds of prefill. A cache miss means the user waits for the full computation. What determines hit vs miss? Whether the KV cache has room for this session's data, whether the eviction policy kept it warm, and whether the router sent this request to the node that has the cache. Those are all systems problems.
4 / 27 The fundamentals

Two stages of inference

Qwen3-4B FP8 on 1x NVIDIA L4 (24 GB)

Prefill is the "thinking" pause. The user waits while the GPU processes the entire input. This determines Time to First Token (TTFT).
Decode is the streaming response. Each new output token reads the entire KV cache to compute attention. Memory-bandwidth-bound, not compute-bound.
Make this concrete. "When you send a prompt to an LLM, two very different things happen. First: prefill. The GPU processes your entire 22,000-token input in parallel. This is compute-heavy, and it's what you experience as the pause before the response starts. That pause is TTFT. Second: decode. The model generates output tokens one at a time. Each new token needs to read the entire KV cache. This is memory-bandwidth-bound." The key insight: these two phases have fundamentally different bottlenecks. Prefill is limited by compute. Decode is limited by memory bandwidth. Same GPU, different constraints.
5 / 27 The insight

The best prefill optimization
is avoiding prefill.

That is what the KV cache does. It stores the result of prefill so you never pay for it twice.

Let this land. The audience just learned that prefill is the expensive phase, the one that determines TTFT, the one that saturates compute. Now tell them: the best optimization is to not do it at all. The KV cache is the mechanism that makes that possible. This frames everything that follows: KV cache is not an optimization detail, it is the central object in inference systems design.
6 / 27 The fundamentals

What is KV cache?

Using Qwen3-4B (FP8) as our running example: 36 layers, 8 KV heads, 128 head_dim

Every token produces a Key and a Value vector at every layer. These are the token's contribution to attention.

The KV cache saves these vectors so they do not need to be recomputed. When the next token arrives, it only computes its own Q, K, V and attends against the cached K and V of all previous tokens.

This is the foundational "what is this thing" slide. Keep the explanation concrete: every token produces a Key ("here's what I contain") and a Value ("here's my actual information"). When a new token arrives, its Query ("what am I looking for?") is compared against all cached Keys to find the relevant context, then pulls the corresponding Values. The KV cache stores these so they don't need to be recomputed. This is the mechanism behind "avoiding prefill" from the previous slide.

9 / 27 The architecture

Disaggregation: separating prefill from decode

  • KV cache becomes the interface between prefill and decode
  • Prefill produces KV, decode consumes it
  • The cache must be placed, routed, evicted, and transferred across nodes
  • Same pattern as database write/read path separation
  • Industry direction: NVIDIA Dynamo, AWS Trainium prefill offload, Cerebras wafer-scale decode
Disaggregation and KV caching are deeply linked. The KV cache IS the interface between prefill and decode. In a disaggregated architecture, it becomes a distributed object: prefill nodes produce it, decode nodes consume it, and the system must decide where to store it, when to evict it, how to transfer it, and which node should handle the next request.
10 / 27 The architecture

Why prefill is the problem

  • Determines TTFT, the latency users actually feel
  • Concurrency cliff is memory-pressure-driven: p99 TTFT jumps 15x while decode barely changes
  • Cache hits eliminate prefill entirely: 388ms down to 57ms
  • Every systems lever applies: routing, placement, eviction, admission control
These optimizations compound. Better routing means fewer cache misses. Fewer misses mean less prefill compute. Less compute means more capacity. Classic distributed systems levers applied to a new domain.
Three data-backed reasons: (1) Prefill determines TTFT. In our L4 benchmarks, the concurrency cliff was about KV cache memory pressure, not compute saturation. (2) Cache hits skip prefill entirely, going from 388ms to 57ms. (3) Every distributed systems primitive applies: cache-aware routing is session affinity, eviction policies are LRU/LFU, tiered storage is write-through caching, admission control is backpressure.
11 / 27 The shared state

How KV cache becomes the bottleneck

Utilization

KV cache reuse avoids redundant prefill compute. A 10K shared system prompt across 50 sessions: 500K tokens computed once instead of 50 times.

Cost

Prefill is the compute-expensive phase: more FLOPs per token than decode, and it blocks the user from seeing any output. Every token you skip via cache hit is GPU time you do not pay for. Under concurrency, the savings compound.

Time to First Token

Users wait for prefill to complete before seeing the first output token. Caching the prefix cuts that wait from seconds to milliseconds. But only if the cache has capacity and the eviction policy cooperates.

Three dimensions where KV cache is the lever. Utilization: without caching, adding users means linearly more prefill work. Cost: prefill is compute-heavy, so avoiding it reduces GPU-hours. TTFT: users experience prefill as the "thinking" pause. When the cache works, TTFT drops dramatically. When it doesn't, users notice immediately.
12 / 27 The shared state

Prefix caching: how it actually works

  • Cross-session: the 10K system prompt is identical for every user. Compute it once, reuse across all sessions.
  • Within-session: turns 1-11 are already cached from earlier in the conversation. Only the new turn needs prefill.
  • Matching is by token block hashing. If two requests share the same token prefix, the engine skips the redundant compute.
The constraint: matching is prefix-only. If tokens diverge at any position, everything after that point must be recomputed. Exact token-for-token match, or nothing.
Two dimensions of caching. Cross-session: the system prompt is identical for every user. Within-session: each turn builds on the previous conversation. The critical limitation is prefix-only matching. This is why workload design matters: put the stable content first (system prompt), variable content last.

From token math to capacity

L4: 24 GB VRAM

// Step 3: concurrency estimate
max_sessions = (kv_capacity - shared_prefix) / per_session_unique

// observed knee: ~14 ✓
How to get available KV memory: vLLM logs it at startup. Look for # GPU blocks in the output. Multiply by block size (16 tokens) and per-token KV bytes to get your capacity in tokens. Or use gpu_memory_utilization (default 0.9) and subtract model weight size. Roughly ~50-55% of raw VRAM ends up available for KV.
Steps 2 and 3. The key practical question is "how do I get the 13 GB number?" Two approaches: vLLM startup logs, or back-of-envelope subtraction. Step 3 is simple division. Gets you within 20% of the observed knee.

22 / 27 The systems lesson

Mixed workloads need per-workload limits

  • Per-session unique context is the dominant variable. A code review agent with 45K unique tokens per session consumes 15x more KV cache than a single-turn QA session.
  • A single global concurrency limit is wrong. You need workload-aware admission control that accounts for each class's cache footprint.
  • The formula applies per workload class: max_sessions = (kv_capacity - shared_prefix) / per_session_unique
If you serve mixed workloads on shared GPUs: one long-context session can evict the KV cache for a dozen short-context sessions. Isolate or prioritize.
The same GPU can handle 168 concurrent single-turn QA sessions or 3 code review sessions. A mixed deployment needs workload-class-aware admission control. One long-context session can thrash the cache for many short-context sessions.
23 / 27 The systems stack

Cache-aware routing

Send requests to the right node

If node A already has the KV cache for session X, route session X's next request to node A. Otherwise, you pay full re-prefill on a cold node. This is session affinity for KV cache.

Local offload: host DRAM and NVMe

GPU memory is scarce and expensive. LMCache moves evicted KV entries to host DRAM or NVMe. Reloading from host memory is faster than recomputing from scratch. A classic tiered cache.

Two concepts within a single node. Routing is session affinity: sticky sessions, but the "session state" is gigabytes of KV cache. Local offload extends the cache hierarchy beyond GPU VRAM.
24 / 27 The systems stack

Tiered KV storage

Remote caching: decouple KV lifetime from GPU lifetime

Push KV state to a networked cache (Redis, Valkey, or a managed service). KV data survives GPU restarts, enables cross-node sharing, and scales capacity independently from compute. Higher latency than local DRAM, but far cheaper than re-prefilling 12,800 tokens.

The full hierarchy

GPU VRAM → Host DRAM → Remote cache → Recompute. Each tier trades latency for capacity. Each boundary is a systems engineering decision: eviction policy, transfer protocol, consistency model.

Remote caching is L3 in the memory hierarchy. A 12,800-token session's KV cache at fp8 is ~900 KB, well within what a networked cache handles at sub-millisecond latency. This decouples KV lifetime from GPU lifetime. Same pattern as session state in web applications, moving from in-process memory to memcached.
25 / 27 The systems stack

How to operate this in production

  1. Estimate your knee before deployment. Three numbers: available KV memory, per-session unique context, KV precision. Get within 20%.
  2. Verify with a concurrency sweep. Run C=1 to 2x your expected peak. Find the cliff. Set your limit below it.
  3. Monitor p95/p99, not averages. Alert on the gap between p50 and p99. When it widens sharply, you have crossed the knee.
  1. Separate online from offline. Offline agents can absorb full prefill cost. Reserve cache resources for online, latency-sensitive workloads.
  2. Benchmark with real data. Synthetic inputs (like hi hi hi repeated 10K times) overstate compression and cache effectiveness. Use real documents with real model weights.
  3. Use the cheapest lever first. KV quantization (fp16 to fp8) is a config change that buys 75% more headroom. Validate output quality on your workload first, then try it before buying more GPUs.
Operational checklist. Each is a systems engineering task, not an ML task. The monitoring point is critical: most inference dashboards show average TTFT, which hides the cliff.

Separate online from offline

Online tier

  • APC enabled
  • Strict concurrency limit at knee
  • p99 SLA enforced
  • Cache-aware routing

Agents, assistants, user-facing apps. Users wait for TTFT.

Offline tier

  • Full prefill acceptable
  • No cache contention
  • Off-peak or separate hardware
  • Throughput-optimized

Scheduled summarization, periodic analysis, proactive monitoring.

The capacity planning question: which workloads should consume scarce cache resources?
Not all workloads need the fastest prefill path. By routing offline agents to a separate tier, you keep KV cache available for online workloads. Same pattern as priority queues in any resource-constrained system.
