4 / 20 The fundamentals

What is KV cache?

Every token that passes through the model produces a Key and a Value vector at every layer. These are the token's contribution to the attention computation.

The KV cache saves these vectors so they do not need to be recomputed. When the next token arrives, it only needs to compute its own Q, K, V and attend against the cached K and V of all previous tokens.

A 22,800-token session on Qwen3-4B with fp8 KV: 36 layers × 8 heads × 128 dims × 2 (K+V) × 22,800 tokens = ~1.6 GB of cached state per session.

This is the foundational "what is this thing" slide. The audience needs to understand what K, V, and Q are before any of the systems implications make sense. Keep the explanation concrete: every token produces a Key ("here's what I contain") and a Value ("here's my actual information"). When a new token arrives, its Query ("what am I looking for?") is compared against all cached Keys to find the relevant context, then pulls the corresponding Values. The KV cache stores these so they don't need to be recomputed. The punchline: for a single agentic session, this cache is 1.6 GB. That's real memory on a 24 GB GPU. Multiply by concurrent sessions and you see why this becomes a memory management problem.
5 / 20 The fundamentals

Two stages of inference

Prefill is the "thinking" pause. The user waits while the GPU processes the entire input. This is what determines Time to First Token (TTFT). For agentic workloads with 22K-token inputs, this can take hundreds of milliseconds to seconds.
Decode is the streaming response. Each new output token reads the entire KV cache (all previous K and V vectors) to compute attention. This is memory-bandwidth-bound, not compute-bound. Users see tokens appearing one at a time.
Make this concrete. "When you send a prompt to an LLM, two very different things happen. First: prefill. The GPU processes your entire 22,000-token input in parallel. All the matrix multiplications, all the attention computations, all at once. This is compute-heavy, and it's what you experience as the pause before the response starts. That pause is TTFT. Second: decode. The model generates output tokens one at a time. Each new token needs to read the entire KV cache to figure out what to say next. This is memory-bandwidth-bound. You see this as the streaming text effect." The key insight for the audience: these two phases have fundamentally different bottlenecks. Prefill is limited by compute (how fast can you do the math). Decode is limited by memory bandwidth (how fast can you read the cache). Same GPU, different constraints.
6 / 20 The fundamentals

Journey of a prompt

Step 1-2: Your prompt arrives and is split into tokens. A 22,800-token agentic session is typical.

Step 3: The engine checks if these tokens match a cached prefix. Hit = skip to the new tokens. Miss = process everything from scratch.

Step 4-5: Prefill runs. The KV cache fills. The first output token appears. This is TTFT: the metric users feel.

Step 5-6: Decode generates tokens one at a time, each reading the full KV cache. The cache is retained for the next turn in the conversation.

Walk through the diagram step by step. This is the slide where the audience builds their mental model of the full request lifecycle. Emphasize the branch at step 3: cache hit vs miss. This single decision point is where all the systems engineering matters. A cache hit means you skip potentially hundreds of milliseconds of prefill. A cache miss means the user waits for the full computation. What determines hit vs miss? Whether the KV cache has room for this session's data, whether the eviction policy kept it warm, and whether the router sent this request to the node that has the cache. Those are all systems problems. Decode is also systems-relevant — memory bandwidth optimization, batching strategies, speculative decoding — but prefill is where the variance lives and where systems engineering has the most dramatic impact on user-visible latency.
7 / 20 The architecture

Disaggregation: separating prefill from decode

Once you separate prefill from decode, KV cache stops being an implementation detail and becomes a distributed systems primitive. It is the output of prefill, the input of decode, and the object that must be placed, routed, evicted, and transferred across nodes.

Why prefill is where systems engineers belong. Prefill is compute-bound and determines TTFT — the latency users feel. Our benchmarks show the concurrency cliff is entirely prefill-driven: p99 TTFT jumps 15x (2.6s to 38.9s) while decode throughput barely changes. A cache hit eliminates prefill entirely. Every systems optimization — routing, placement, eviction, admission control — directly reduces prefill cost. These are classic distributed systems levers applied to a new domain.

This is happening in production: NVIDIA Dynamo, AWS Trainium prefill offload, Cerebras wafer-scale decode. The pattern is the same one distributed systems solved for databases: separate the write-heavy path (prefill computes KV) from the read-heavy path (decode reads KV) and optimize each independently.

This is the central connection of the talk, so take time here. Disaggregation and KV caching are not separate topics — they are deeply linked. The KV cache IS the interface between prefill and decode. In a monolithic engine, the KV cache is an in-memory data structure on one GPU. In a disaggregated architecture, it becomes a distributed object: prefill nodes produce it, decode nodes consume it, and the system must decide where to store it, when to evict it, how to transfer it, and which node should handle the next request for a given session. Why does prefill matter more to systems engineers? Three reasons backed by our data: (1) Prefill determines TTFT, the metric users actually experience. In our L4 benchmarks, the concurrency cliff was entirely about prefill — p99 TTFT jumped 15x when the KV cache filled, while per-token decode latency was essentially constant. The cliff is a prefill problem. (2) Cache hits skip prefill entirely — going from 388ms to 57ms. No systems optimization can make decode 6x faster, but a cache hit gives you that for free on the prefill side. (3) Every distributed systems primitive applies to prefill: cache-aware routing is session affinity. Eviction policies are LRU/LFU. Tiered storage (LMCache) is write-through caching. Admission control is backpressure. These are problems systems engineers have solved before. The hardware is GPUs instead of servers, but the abstractions are identical. The fun part: these optimizations compound. Better routing means fewer cache misses. Fewer misses mean less prefill compute. Less compute means more capacity for concurrent sessions. More sessions mean the system pays for itself faster. This is why the best prefill service is a distributed systems service — and why systems engineers should be excited, not intimidated.
8 / 20 The shared state

KV cache makes or breaks everything

Utilization

KV cache reuse avoids redundant prefill compute. A 10K shared system prompt across 50 sessions: 500K tokens computed once instead of 50 times.

Cost

Prefill is the expensive phase. Every token you skip via cache hit is GPU time you do not pay for. Under concurrency, the savings compound.

Time to First Token

Users wait for prefill to complete before seeing the first output token. Caching the prefix cuts that wait from seconds to milliseconds. But only if the cache has capacity and the eviction policy cooperates.

Three concrete dimensions where KV cache is the lever. Utilization: without caching, adding concurrent users means linearly more prefill work. Cost: prefill is compute-heavy, so avoiding it directly reduces GPU-hours. TTFT: users experience prefill as the "thinking" pause before the response starts streaming. When the cache works, TTFT drops dramatically. When it doesn't (cache miss, evicted session), users notice immediately. This is why KV cache management is the central systems challenge in inference.
9 / 20 The shared state

Prefix caching: how it actually works

Inference engines divide tokens into blocks and hash them. If a new request shares the same token prefix as a previous one, the engine reuses the cached KV computations instead of recomputing them.

The constraint: matching is prefix-only. If tokens diverge at any position, everything after that point must be recomputed. Exact token-for-token match, or nothing.
Keep this general. The audience doesn't need to know the difference between vLLM's hash chain and SGLang's radix tree. What they need is the mental model: tokens are grouped into blocks, hashed with their position, and if two requests share the same prefix, the engine skips the redundant compute. The critical limitation is prefix-only. Rearranging content, inserting a different document in the middle, or changing a single token early in the prompt invalidates everything after it. This is why workload design matters: put the stable content first (system prompt), variable content last.

11 / 20 The experiment

The concurrency cliff

p99 TTFT jumps from 2.6 seconds to 38.9 seconds between C=12 and C=14. Not a single re-prefill — cascading evictions under contention, with queuing delays compounding.

Let this chart land. Below 12 sessions, everything fits in the KV cache. At 14, the cache fills. LRU eviction kicks in, and it is not just one re-prefill: multiple sessions thrash for the same cache space, evicting each other in a cascade. Each evicted session queues behind others waiting for prefill compute. A single 12.8K re-prefill costs ~220ms, but when many sessions are evicted simultaneously, queuing and scheduling delays compound to tens of seconds. The p50 barely moves because lucky requests still hit warm cache. This is a capacity-driven cliff, not a compute cliff. The GPU is not saturated; the memory is full.

13 / 20 The systems lesson

The knee is predictable from first principles

max_sessions = (kv_memory - shared_prefix) / per_session_unique

// L4 with fp8 KV, Qwen3-4B:
(178K tokens - 10K shared) / 12.8K per session = ~13 sessions
// observed: ~14 ✓

Three variables:

  1. Available KV memory = VRAM - model - CUDA graphs - activations (~50% of raw VRAM)
  2. Per-session unique context = your workload's per-session token count minus shared prefix
  3. KV precision = bytes per element (halving precision roughly doubles capacity)
This is where systems engineers feel at home. The concurrency cliff is a capacity problem with a closed-form estimate. The formula gets you within 20%. It tells you whether you're looking at 5 sessions or 50, which is the critical capacity planning question.
14 / 20 The systems lesson

Small systems choices, big performance changes

Changing one config flag (kv_cache_dtype) shifts the concurrency knee from 8 sessions to 14 to 23. Same GPU. Same model. Same workload.

4-bit caveat: KV quantization below fp8 can measurably impact output quality on reasoning and long-context tasks. fp8 is generally safe; 4-bit requires quality validation for your workload.
This is the "live tuning" slide. One config change, three different concurrency ceilings. fp16 to fp8 doubles capacity and moves the knee from ~8 to ~14. fp8 to 4-bit extends it to ~23. You didn't retrain the model or buy a bigger GPU. You changed a config flag and got 75% more concurrent sessions. This is a systems lever, not an ML lever. Important caveat for 4-bit: our fp8 tests showed no quality degradation on Qwen3-4B, but sub-8-bit KV quantization (4-bit, 2-bit) can introduce measurable output quality loss, especially on multi-step reasoning and long-context tasks. Always validate quality metrics before deploying aggressive quantization in production. fp8 is the safe default; 4-bit is a tradeoff you need to evaluate per workload.

A session evicted at turn 3 only costs ~3,000 tokens to re-prefill. Worst case forces every miss at maximum depth. Real traffic has a distribution of miss depths, which is why realistic stays much closer to best.

17 / 20 The systems stack

Cache-aware routing and LMCache

Routing: send requests to the right node

If node A already has the KV cache for session X, route session X's next request to node A. Otherwise, you pay full re-prefill on a cold node. This is session affinity for KV cache.

LMCache: offload KV to host memory or storage

GPU memory is scarce and expensive. LMCache moves evicted KV entries to host DRAM or NVMe, creating a tiered cache hierarchy. Reloading from host memory is faster than recomputing from scratch.

The stack is converging

vLLM + cache-aware routing + LMCache + disaggregated prefill. Each layer is a systems problem: scheduling, placement, eviction, transfer.

Walk through the emerging systems stack. Cache-aware routing is sticky sessions for GPUs. LMCache is a write-through cache hierarchy like a database buffer pool. Disaggregated prefill is compute/storage separation. Each layer maps directly to a distributed systems primitive the audience already knows.

How to operate this in production

  1. Estimate your knee before deployment. Three numbers: available KV memory, per-session unique context, KV precision. Get within 20%.
  2. Verify with a concurrency sweep. Run C=1 to 2x your expected peak. Find the cliff. Set your limit below it.
  3. Monitor p95/p99, not averages. Alert on the gap between p50 and p99. When it widens sharply, you have crossed the knee.
  1. Separate background from interactive. Background agents can absorb full prefill cost. Reserve cache resources for latency-sensitive workloads.
  2. Benchmark with real data. Synthetic inputs (like hi hi hi repeated 10K times) overstate compression and cache effectiveness. Use real documents with real model weights.
  3. Use the cheapest lever first. KV quantization (fp16 to fp8) is a config change that buys 75% more headroom. Try it before buying more GPUs.
Operational checklist. These are the concrete things to do on Monday morning. Each is a systems engineering task, not an ML task. The monitoring point is critical: most inference dashboards show average TTFT, which hides the cliff. Point 5 about benchmarking: many KV cache benchmarks use synthetic "hi hi hi" inputs with only 2 unique tokens. Real medical text has 14% token diversity. Real model weights produce high-entropy KV activations that compress poorly. Your capacity plans will be wrong if you benchmark with synthetic data.

Separate background from interactive

Interactive tier

  • APC enabled
  • Strict concurrency limit at knee
  • p99 SLA enforced
  • Cache-aware routing

Agents, assistants, user-facing apps. Users wait for TTFT.

Background tier

  • Full prefill acceptable
  • No cache contention
  • Off-peak or separate hardware
  • Throughput-optimized

Scheduled summarization, periodic analysis, proactive monitoring.

The capacity planning question: which workloads should consume scarce cache resources?
Not all workloads need the fastest prefill path. Background agents can trade latency for cost. By routing them to a separate tier, you keep the KV cache available for interactive workloads. This is the same pattern as priority queues in any resource-constrained system. Only interactive sessions count toward the knee.
