2 / 20 Part I: Why KV Cache

The case against KV caching

  • Bounded single-request savings. 4% for short inputs, 24% for long inputs. Not transformative in isolation.
  • Enormous memory cost. KV cache competes with model weights and activations for the same VRAM.
  • Narrow sharing model. Prefix caching requires exact token-for-token prefix match. Any divergence breaks the chain.
Start with the skeptic's view. Single-request latency improvement is bounded by the input/output ratio: a 1K-input/512-output request saves ~4% of total time (135ms TTFT out of 2.5s total), while a 16K-input/512-output request saves ~24%. Memory is scarce on GPUs, and prefix caching only works for exact prefix matches, so RAG with variable retrieval chunks gets no benefit.

4 / 20 Part I: Why KV Cache

The real benefit is system-level

Single-request savings are modest. But under concurrency, avoiding repeated prefill reduces queue interference.

Without KV cache: Every request prefills from scratch. Prefill is compute-heavy, blocks decode for other requests. Head-of-line blocking under load.
With KV cache: Shared prefixes computed once. Subsequent requests skip to their unique suffix. GPU spends more time on decode, less on redundant prefill.
This is the argument that makes KV caching go from "nice optimization" to "critical infrastructure." At scale, the interference cost of redundant prefill dominates. A 10K system prompt shared by 100 concurrent sessions means the difference between prefilling 1M tokens vs 10K tokens of shared content. The GPU time freed up directly translates to lower tail latency for all requests.
5 / 20 Part I: Why KV Cache

Disaggregation changes everything

Prefill and decode have different compute profiles. Separating them turns KV cache from an implementation detail into a distributed systems primitive.

Production examples: NVIDIA Dynamo, AWS Trainium prefill offload, Cerebras wafer-scale decode.

Post 2's central thesis. Prefill is compute-bound (high arithmetic intensity), decode is memory-bandwidth-bound (one token at a time, reading all KV entries). Colocating them forces hardware compromise. Disaggregation lets you put prefill on high-FLOPS hardware and decode on high-bandwidth/large-memory hardware. The KV cache becomes the object that moves across the boundary, requiring transfer, placement, eviction, and routing policies. This is the same structural shift as Snowflake separating storage and compute.

The economics are moving fast. Model architecture innovations are shrinking the KV cache by orders of magnitude. MLA compresses the key-value representation itself. CSA uses sparse attention to avoid materializing most of the cache. DeltaNet replaces per-token KV with fixed-size recurrent state. And quantization works on any existing model without retraining. These compound: a model with MLA + 4-bit quantization might have 100x smaller KV cache than standard MHA at fp16.
7 / 20 Part II: How It Works

vLLM: hash chain prefix caching

  • Block size: 16 tokens (CUDA). Each block hashed with all preceding hashes.
  • O(1) lookup per block via hash table. O(1) eviction via LRU linked list.
  • Limitation: Block-aligned matching only. 15/16 matching tokens = complete hash miss.
Post 3 deep dive. vLLM's Automatic Prefix Caching (APC) uses a hash chain: block N's hash includes the hash of block N-1, which includes N-2, and so on. This creates a content-addressable fingerprint that encodes the full token history up to that point. Two requests with identical prefixes will produce identical hashes for all shared blocks. But any divergence at any position breaks the chain for every subsequent block. The matching is also block-aligned at 16-token boundaries, so there's up to 15 tokens of wasted overlap. In practice, the overhead is small for long prefixes (~0.4% for a 4K prefix).

Both are still prefix-bound. Neither handles non-prefix content reuse.
SGLang takes a different approach: a radix tree (compressed trie) indexed by token sequences. This allows token-level matching at arbitrary boundaries, not just 16-token blocks. It naturally handles branching conversations: different turns from the same prefix share the tree structure. HiRadix extends this to multi-tier storage (GPU, host memory, distributed). The tradeoff: higher CPU overhead from tree traversal (O(prefix length) vs O(1)), but higher effective hit rates for variable-length turns.

  • Compression: Repetitive tokens compress well. Real activations from trained weights produce high-entropy floats.

The distortion matters because real trained model weights (Qwen3-8B-FP8) produce high-entropy floating-point KV activations that compress poorly. Dummy weights show great compression ratios; real weights show modest gains. We built a 30-document corpus (15 medical, 15 legal, 300K tokens total) with real Qwen3-8B-FP8 weights and noise injection. The corpus is ~700x more diverse than synthetic baselines.

11 / 20 Part III: Concurrency

The concurrency cliff

p99 TTFT jumps from 2.6s to 38.9s between C=12 and C=14. A 15x increase in two sessions.

The central finding of Post 5. Below 12 concurrent sessions, everything fits in the L4's KV cache. APC caches the system prompt once and extends each session's turn history incrementally. At 14 sessions, the KV cache fills. vLLM starts evicting blocks via LRU, and evicted sessions must re-prefill their entire turn history on the next request. The p99 explodes because it captures the unlucky requests that hit evicted sessions. The p50 barely moves because lucky requests still hit warm cache entries.

Each precision level shifts the knee proportionally. fp16 to fp8 doubles token capacity (89K to 178K) and shifts the knee from ~8c to ~14c (1.75x, slightly less than 2x due to overhead). fp8 to tq4 adds another ~55% (178K to 275K), shifting the knee to ~23c. The tq4 estimate accounts for lower gpu_memory_utilization (0.8 vs 0.9 for fp8) and ~0.53 bytes effective per element (quantization metadata overhead). Quantization is a concurrency multiplier: same hardware, same workload, more sessions.

Miss depth matters. Realistic misses happen at any turn depth (turn 3 = ~3K tokens). Worst forces every miss at peak depth (turn 12 = ~12.8K tokens). That is why realistic stays much closer to best than to worst.
We ran two controlled bounds alongside the realistic workload. Best case: every request hits the same cached content, only ~200 unique tokens need prefill. Worst case: a bust prefix breaks APC for user context, but the system prompt still hits the cache. The big difference is that worst forces all misses at peak session depth (turn 12, ~12,800 tokens), while realistic misses can happen at any turn. A turn 3 eviction only costs ~3,000 tokens of re-prefill. The best case shows the L4 handles over 48 concurrent sessions within 2s p99 if perfectly cached. The concurrency limit is a memory capacity problem, not a compute limitation.

The math: why the knee is where it is

GPU memory budget

L4: 24 GB VRAM
Available for KV cache: ~13 GB (after model, CUDA graphs, activations)
= ~50% of raw VRAM, typical for inference

Per-token KV size (Qwen3-4B)

36 layers x 8 KV heads x 128 head_dim x 2 (K+V)
FP16: 147,456 bytes/token -> ~89K tokens in 13 GB
FP8: 73,728 bytes/token -> ~178K tokens in 13 GB

Capacity with APC

system prompt: 10K tokens (stored once, shared)
per session: ~12.8K tokens at peak (turn 12)

FP8: (178K - 10K) / 12.8K = ~13 sessions observed: ~14 ✓
FP16: (89K - 10K) / 12.8K = ~6 sessions observed: ~8 ✓
The knee is predictable from first principles. You need three numbers: available KV memory (typically ~50% of VRAM after model, CUDA graphs, and activations), per-token KV size (determined by model architecture and precision), and per-session unique context (total tokens at peak depth minus shared prefix). The formula is: (token_capacity - shared_prefix) / per_session_unique = max_sessions. The slight discrepancy between predicted and observed (13 vs 14 for fp8) comes from session depth staggering: not all sessions are at turn 12 simultaneously, and vLLM can partially cache sessions.
16 / 20 Part IV: Deployment

Find your knee

max_sessions = (kv_capacity - shared_prefix) / per_session_unique

Three variables to estimate

  1. Available KV memory = total VRAM - model - CUDA graphs - activations
    Typically ~50% of raw VRAM for inference
  2. Per-session unique context = tokens at peak depth - shared prefix
    Your workload determines this. Agentic coding: ~12.8K. Single-turn QA: ~2K.
  3. KV precision = bytes per element
    Each halving roughly doubles capacity, shifts knee right.
Estimate before deployment, verify with a concurrency sweep. The formula gets you within 20%. The sweep confirms bimodal behavior at the boundary.
This is the actionable takeaway. Before deploying, calculate your expected knee using these three numbers. Then verify with a concurrency sweep like we did (C=1 to 48). The formula is within ~20% because it doesn't account for session depth staggering, block fragmentation, or the specific eviction behavior of your inference engine. But it tells you whether you're looking at 5 sessions or 50, which is the critical capacity planning question.

The most important operational takeaway. After the concurrency cliff, the latency distribution becomes bimodal. Half your users get fast responses (cache hit), half get catastrophically slow responses (cache miss + full re-prefill). The mean and median hide this completely. If your monitoring only tracks average TTFT, you won't know you've crossed the cliff until users start complaining. Track p95 and p99, and set alerts on the gap between p50 and p99. When that gap widens sharply, you've hit the knee.
19 / 20 Part IV: Deployment

Separate background from interactive

Not all workloads need the fastest prefill path. Background agents (scheduled summarization, periodic analysis, proactive monitoring) are not latency-sensitive.

Let background work absorb full prefill cost. Reserve KV cache resources for workloads that need low TTFT: interactive agents, real-time assistants, user-facing applications.

This is a scheduling distinction, not a caching one

  • Interactive tier: APC enabled, strict concurrency limit at knee, p99 SLA enforced.
  • Background tier: Full prefill OK, no cache contention, can run during off-peak or on separate hardware.

The capacity planning question is: which workloads should consume scarce cache resources?

From Post 1. There's a growing class of agentic workloads that run on a schedule rather than in response to a user waiting. These can trade latency tolerance for cost and throughput. By routing background workloads to a separate tier (or scheduling them during off-peak), you keep the KV cache available for interactive workloads that actually need low TTFT. This is the same pattern as priority queues in any resource-constrained system. The distinction matters for capacity planning because it determines your effective concurrency limit: only interactive sessions count toward the knee.
20 / 20 Summary

Key takeaways

  1. KV cache matters at system level, not single-request level. Concurrency is where it pays off.
  2. Disaggregation makes KV cache a first-class primitive with placement, eviction, and routing requirements.
  3. Both vLLM and SGLang are prefix-bound. Exact prefix match or nothing. Design workloads accordingly.
  4. Benchmark with real data. Synthetic inputs overestimate compression and cache effectiveness.
  5. The concurrency cliff is sharp and predictable. Estimate the knee from memory budget, verify with a sweep.
  6. Quantization is a concurrency multiplier. fp8 = 1.75x more sessions. Each halving of precision shifts the knee right.
  7. Monitor p99, not averages. After the cliff, latency is bimodal and the mean is fiction.
