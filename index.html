4 / 24 Steps in inference

Journey of a prompt

Build the diagram step by step. Phase 1: prompt arrives. Phase 2: tokenized into 22,800 tokens. Phase 3: engine checks cache — MISS means process everything from scratch. Phase 4: prefill runs, KV cache fills, first token appears (TTFT). Phase 5: decode loop generates tokens one at a time reading the full KV cache, response completes — engine tries to retain cache but eviction pressure can force re-prefill. Phase 6: reveal the HIT path — skip prefill entirely, go straight to cached KV state. This is the punchline: the best prefill optimization is avoiding prefill.
5 / 24 Journey of a prompt

Inside NVIDIA Dynamo: the request lifecycle

Walk through the Dynamo lifecycle. Phase 1: request arrives at the Router, which tokenizes the input and queries the KVIndexer (a radix tree tracking cached blocks per worker) to find the prefill node with the most KV cache overlap. Phase 2: the prefill node computes KV vectors and returns transfer metadata. The router selects a decode node and forwards the request. NIXL performs direct GPU-to-GPU KV transfer via RDMA or NVLink, zero-copy. Phase 3: the decode node generates tokens one at a time, streaming back through the router. The router is the brain, the KV cache is the shared state, and NIXL is the data plane.
6 / 24 Inference pipeline

Two stages of inference

Qwen3-4B FP8 on 1x NVIDIA L4 (24 GB)

Prefill is the "thinking" pause. The GPU processes the entire input. This determines TTFT.
Decode is the streaming response. Each token reads the entire KV cache. Memory-bandwidth-bound.
Make this concrete. "When you send a prompt to an LLM, two very different things happen. First: prefill. The GPU processes your entire 22,000-token input in parallel. This is compute-heavy, and it's what you experience as the pause before the response starts. That pause is TTFT. Second: decode. The model generates output tokens one at a time. Each new token needs to read the entire KV cache. This is memory-bandwidth-bound." The key insight: these two phases have fundamentally different bottlenecks. Prefill is limited by compute. Decode is limited by memory bandwidth. Same GPU, different constraints.
7 / 24 Optimizing prefill stage

Minimize generating prefill.

KV Cache retains the prefill results to help avoid paying for the prefill compute on the next turn.

Let this land. The audience just learned that prefill is the expensive phase, the one that determines TTFT, the one that saturates compute. Now tell them: the best optimization is to not do it at all. The KV cache is the mechanism that makes that possible. This frames everything that follows: KV cache is not an optimization detail, it is the central object in inference systems design.
8 / 24 The fundamentals

What is KV cache?

Using Qwen3-4B (FP8) as our running example: 36 layers, 8 KV heads, 128 head_dim

Every token produces a Key and a Value vector at every layer. These are the token's contribution to attention. The KV cache saves these vectors so they do not need to be recomputed. When the next token arrives, it only computes its own Q, K, V and attends against the cached K and V of all previous tokens. This is the mechanism behind "avoiding prefill" from the previous slide.

72 KB per token sounds small, but 22,800 tokens at 72 KB each is 1.6 GB. On a 24 GB L4, that is nearly 7% of total VRAM for a single session. 10 / 24 The architecture

Disaggregation: separating prefill from decode

  • KV cache becomes the interface between prefill and decode
  • Prefill produces KV, decode consumes it
  • The cache must be placed, routed, evicted, and transferred across nodes
  • Same pattern as database write/read path separation
  • Industry direction: NVIDIA Dynamo, AWS Trainium prefill offload, Cerebras wafer-scale decode
Disaggregation and KV caching are deeply linked. The KV cache IS the interface between prefill and decode. In a disaggregated architecture, it becomes a distributed object: prefill nodes produce it, decode nodes consume it, and the system must decide where to store it, when to evict it, how to transfer it, and which node should handle the next request.

Disaggregation: the data

Prefill is compute-bound (90-95% GPU utilization). Decode is memory-bound (20-40%). Running both on the same GPU wastes whichever resource isn't the bottleneck.

The pattern: prefill and decode have fundamentally different hardware profiles. Separating them lets each phase run on optimal hardware, and lets each pool scale independently.
Caveat: disaggregation adds KV transfer latency (~27ms over InfiniBand). Short prompts with high cache hit rates can see 20-30% degradation. The gains scale with input length and output-to-input ratio.
Every major inference system has converged on disaggregation: Dynamo, llm-d, SGLang, vLLM, Mooncake, DistServe. The Splitwise numbers are the cleanest: prefill achieves 90-95% GPU compute utilization at 200-400 ops/byte, while decode gets only 20-40% at 60-80 ops/byte. Running both on the same GPU wastes compute during decode and bandwidth during prefill. The 27ms KV transfer cost of disaggregation replaces 200-500ms of queuing delay from prefill-decode interference. Not all workloads benefit equally, but for long-context and reasoning workloads, the data is unambiguous.
12 / 24 The shared state

Prefix caching: how it actually works

  • Cross-session: the 10K system prompt is identical for every user. Compute it once, reuse across all sessions.
  • Within-session: turns 1-11 are already cached from earlier in the conversation. Only the new turn needs prefill.
  • Matching is by token block hashing. If two requests share the same token prefix, the engine skips the redundant compute.
The constraint: matching is prefix-only. If tokens diverge at any position, everything after that point must be recomputed. Exact token-for-token match, or nothing.
Two dimensions of caching. Cross-session: the system prompt is identical for every user. Within-session: each turn builds on the previous conversation. The critical limitation is prefix-only matching. This is why workload design matters: put the stable content first (system prompt), variable content last.

Predict the concurrency knee

// Step 1: KV memory per token (from config.json)
per_token_kv = 36 layers × 8 kv_heads × 128 head_dim × 2(K,V) × 1B (fp8) = 72 KB/token

// Step 2: available KV capacity
L4: 24 GB - ~4 GB weights - ~5 GB overhead - ~2 GB reserved ≈ ~13 GB for KV ≈ 178K tokens

// Step 3: concurrency estimate
max_sessions = (178K - 10K shared) / 12.8K per session = ~13 sessions // observed: ~14 ✓
How to get available KV memory: vLLM logs # GPU blocks at startup. Multiply by block size (16 tokens) × per-token KV bytes. Or: ~50-55% of raw VRAM ends up available for KV.
Three steps, one slide. Step 1: per-token KV from config.json (GQA gives 8 heads instead of 32). Step 2: subtract weights + overhead from VRAM. Step 3: simple division. Gets within 20% of observed knee. The vLLM startup log is the practical way to get the exact number.

One config flag, 75% more headroom

Qwen3-4B on 1x NVIDIA L4 (24 GB), vLLM 0.20.2

Changing one config flag (kv_cache_dtype) shifts the concurrency knee from 8 sessions to 14 to 23. Same GPU. Same model. Same workload.

4-bit caveat: KV quantization below fp8 can measurably impact output quality on reasoning and long-context tasks. fp8 is generally safe; 4-bit requires quality validation for your workload.
One config change, three different concurrency ceilings. fp16 to fp8 doubles capacity and moves the knee from ~8 to ~14. fp8 to 4-bit extends it to ~23. This is a systems lever, not an ML lever.

The cost of cache misses

57ms with perfect caching. Prefill cost nearly vanishes. The realistic 388ms per turn is not a compute limitation. It is the cost of per-session context uniqueness: the tokens that can never be shared.
Miss depth matters. Realistic misses happen at any turn depth (turn 3 = ~3K tokens). Worst forces every miss at peak depth (turn 12 = ~12.8K tokens). Real traffic is much kinder than worst case.
The gap between best and realistic represents the fundamental cost of per-session uniqueness. The gap between realistic and worst shows how much eviction patterns matter.

21 / 24 The systems stack

Cache-aware routing

Send requests to the right node

If node A already has the KV cache for session X, route session X's next request to node A. Otherwise, you pay full re-prefill on a cold node. This is session affinity for KV cache.

Local offload: host DRAM and NVMe

GPU memory is scarce and expensive. LMCache moves evicted KV entries to host DRAM or NVMe. Reloading from host memory is faster than recomputing from scratch. A classic tiered cache.

Two concepts within a single node. Routing is session affinity: sticky sessions, but the "session state" is gigabytes of KV cache. Local offload extends the cache hierarchy beyond GPU VRAM.
22 / 24 The systems stack

Tiered KV storage

Remote caching: decouple KV lifetime from GPU lifetime

Push KV state to a networked cache (Redis, Valkey, or a managed service). KV data survives GPU restarts, enables cross-node sharing, and scales capacity independently from compute. Higher latency than local DRAM, but far cheaper than re-prefilling 12,800 tokens.

The full hierarchy

GPU VRAM → Host DRAM → Remote cache → Recompute. Each tier trades latency for capacity. Each boundary is a systems engineering decision: eviction policy, transfer protocol, consistency model.

Remote caching is L3 in the memory hierarchy. A 12,800-token session's KV cache at fp8 is ~900 KB, well within what a networked cache handles at sub-millisecond latency. This decouples KV lifetime from GPU lifetime. Same pattern as session state in web applications, moving from in-process memory to memcached.
23 / 24 The systems stack

Operating rules

  1. Estimate your knee before deployment. Three numbers: available KV memory, per-session unique context, KV precision. Get within 20%.
  2. Verify with a concurrency sweep. Run C=1 to 2x your expected peak. Find the cliff. Set your limit below it.
  3. Monitor p95/p99, not averages. Alert on the gap between p50 and p99. When it widens sharply, you have crossed the knee.
  1. Separate online from offline. Offline agents can absorb full prefill cost. Reserve cache resources for online, latency-sensitive workloads.
  2. Benchmark with real data. Synthetic inputs (like hi hi hi repeated 10K times) overstate compression and cache effectiveness. Use real documents with real model weights.
  3. Use the cheapest lever first. KV quantization (fp16 to fp8) is a config change that buys 75% more headroom. Validate output quality on your workload first, then try it before buying more GPUs.
Operational checklist. Each is a systems engineering task, not an ML task. The monitoring point is critical: most inference dashboards show average TTFT, which hides the cliff.
