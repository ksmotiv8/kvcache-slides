4 / 20 The architecture

Two workloads hiding inside one GPU

Prefill

Process the entire input at once. Thousands of tokens in parallel.

  • Compute-bound. High arithmetic intensity.
  • Bursty. Arrives all at once, variable size.
  • Blocking. Must finish before first output token.

Decode

Generate output one token at a time. Sequential, stateful.

  • Memory-bandwidth-bound. Reads all KV entries per token.
  • Latency-sensitive. Users wait for each token.
  • Stateful. Depends on the KV cache from prefill.
The first systems insight: these have fundamentally different resource profiles. Colocating them forces a compromise that hurts both.
This is the foundational insight. Prefill is like a batch ETL job: compute-heavy, parallelizable, variable size. Decode is like a streaming service: latency-sensitive, sequential, steady-state. Putting them on the same GPU is like running your batch jobs on your latency-sensitive serving tier. Prefill can cause head-of-line blocking: a large prefill starves decode for other requests, spiking their inter-token latency. This is a scheduling problem, not a GPU problem.

6 / 20 The architecture

Disaggregation: separating prefill from decode

Once you separate prefill from decode, KV cache stops being an implementation detail and becomes a distributed systems primitive. It needs placement, eviction, routing, and lifecycle management.

This is happening in production: NVIDIA Dynamo, AWS Trainium prefill offload, Cerebras wafer-scale decode.

Draw the parallel to storage/compute separation in data systems. Snowflake separated storage and compute, and suddenly the storage layer needed its own scheduling, caching, and routing policies. The same thing is happening here. KV cache is the state that bridges the prefill/decode boundary. When those are on different hardware, the KV cache needs transfer protocols, placement policies (which GPU has the cache for this session?), eviction strategies, and routing (send the next request to the node that already has the KV cache). These are textbook distributed systems problems.
7 / 20 The shared state

KV cache makes or breaks everything

Utilization

KV cache reuse avoids redundant prefill compute. Without it, the GPU re-processes the same tokens over and over. With it, shared system prompts are computed once and reused across sessions.

Cost

Prefill is the expensive phase. Every token you skip via cache hit is GPU time you do not pay for. At scale, a 10K shared prefix across 100 sessions is the difference between prefilling 1M tokens vs 10K.

Time to First Token

Users wait for prefill to complete before seeing the first output token. Caching the prefix cuts that wait from seconds to milliseconds. But only if the cache has capacity and the eviction policy cooperates.

Three concrete dimensions where KV cache is the lever. Utilization: without caching, adding concurrent users means linearly more prefill work. Cost: prefill is compute-heavy, so avoiding it directly reduces GPU-hours. TTFT: users experience prefill as the "thinking" pause before the response starts streaming. When the cache works, TTFT drops dramatically. When it doesn't, users notice. This is why KV cache management is the central systems challenge in inference.
8 / 20 The shared state

Prefix caching: how it actually works

Inference engines like vLLM divide tokens into blocks and hash them. If a new request shares the same token prefix as a previous one, the engine reuses the cached KV computations instead of recomputing them.

The constraint: matching is prefix-only. If tokens diverge at any position, everything after that point must be recomputed. Exact token-for-token match, or nothing.
Keep this general. The audience doesn't need to know the difference between vLLM's hash chain and SGLang's radix tree. What they need is the mental model: tokens are grouped into blocks, blocks are hashed with their position history, and if two requests share the same prefix, the engine skips the redundant compute. The critical limitation is that it's prefix-only. Rearranging content, inserting a different document in the middle, or changing a single token early in the prompt invalidates everything after it. This is why workload design matters: put the stable content first (system prompt), variable content last.
9 / 20 The shared state

The benefit is not per-request. It is under load.

4-24%

Single-request latency savings. Bounded by input/output ratio. Modest.

10-100x

Prefill compute avoided under concurrency. A 10K shared prefix across 50 sessions: 500K tokens computed once instead of 50 times.

The system-level insight: avoiding redundant prefill reduces GPU queue depth, which reduces tail latency for all requests. One user's cache hit improves everyone's experience.
This is the turn where the audience should see why this matters. In isolation, KV caching saves 4-24% of one request's time. Under concurrency, the GPU queue effect dominates: every request that skips prefill frees GPU cycles for decode, reducing latency for all other requests. It's the same dynamic as connection pooling in databases: the per-connection overhead looks small, but the queue effects under load are what break your p99. KV caching is the same pattern applied to GPU compute.

11 / 20 The experiment

The concurrency cliff

p99 TTFT jumps from 2.6 seconds to 38.9 seconds between C=12 and C=14. A 15x increase in two sessions.

Let this chart land. The cliff is dramatic. Below 12 sessions, everything fits in the KV cache. vLLM caches the system prompt once, extends each session's turn history incrementally, and p99 stays under 2.6 seconds. At 14 sessions, the KV cache fills. LRU eviction kicks in, and evicted sessions must re-prefill their entire turn history. This is a capacity-driven cliff, not a compute cliff. The GPU is not saturated; it's the memory that's full. Point at the red line: that's the user experience for the unlucky 1%.

This is the systems engineering insight that matters most in practice. After the cliff, the latency distribution is bimodal. Some requests land on cached sessions and complete fast. Others land on evicted sessions and wait 30-50 seconds. If your dashboard shows average TTFT, it will show ~5 seconds at C=14. That average is fiction: nobody actually experiences 5 seconds. Half experience 1.3 seconds, the other half experience 39 seconds. This is the same problem as measuring average latency in a system with circuit breakers: the average hides the failure mode.
13 / 20 The systems lesson

The knee is predictable from first principles

max_sessions = (kv_memory - shared_prefix) / per_session_unique

// L4 with fp8 KV, Qwen3-4B:
(178K tokens - 10K shared) / 12.8K per session = ~13 sessions
// observed: ~14 ✓

Three variables:

  1. Available KV memory = VRAM - model - CUDA graphs - activations (~50% of raw VRAM)
  2. Per-session unique context = your workload's per-session token count minus shared prefix
  3. KV precision = bytes per element (halving precision roughly doubles capacity)
This is where systems engineers feel at home. The concurrency cliff is not mysterious: it's a capacity problem with a closed-form estimate. You need three numbers, all knowable before deployment. Available KV memory is typically ~50% of raw VRAM after accounting for model weights, CUDA graphs, and activation memory. Per-session unique context comes from your workload profile. KV precision is a deployment choice. The formula gets you within 20% because it doesn't account for session depth staggering and block fragmentation, but it tells you the order of magnitude: are you looking at 5 sessions or 50?
14 / 20 The systems lesson

Small systems choices, big performance changes

Changing one config flag (kv_cache_dtype) shifts the concurrency knee from 8 sessions to 14 to 23. Same GPU. Same model. Same workload.

This is the "live tuning" slide from the abstract. One systems-level config change — KV cache precision — shifts the entire cliff. fp16 to fp8 doubles token capacity and moves the knee from ~8 to ~14 (1.75x). fp8 to 4-bit quantization extends it to ~23. This is what I mean by "small systems choices change real performance." You didn't retrain the model. You didn't buy a bigger GPU. You changed a config flag and got 75% more concurrent sessions. This is a systems lever, not an ML lever.

The 57ms best case is striking. With perfect caching, this GPU handles 48+ concurrent sessions within a 2s p99 SLA. The ~6-session realistic ceiling is not a GPU compute limitation. It is the cost of per-session context uniqueness.
Miss depth distribution matters. In realistic traffic, misses happen at any turn depth. Turn 3 miss: re-prefill ~3K tokens. Turn 12 miss: ~12.8K tokens. Worst case forces every miss at maximum depth. Real traffic is much kinder.
This is the capacity planning insight. The gap between best (57ms) and realistic (388ms) represents the fundamental cost of per-session uniqueness. APC eliminates the system prompt cost, but each session has ~12,800 unique tokens that must be computed at least once. The gap between realistic and worst shows how much the eviction pattern matters. Worst case forces every miss at peak session depth (turn 12), but real traffic has a distribution of miss depths. A session evicted at turn 3 only costs ~3,000 tokens to re-prefill. This is why realistic stays much closer to best than to worst on an absolute scale.

17 / 20 The systems stack

Cache-aware routing and LMCache

Routing: send requests to the right node

If node A already has the KV cache for session X, route session X's next request to node A. Otherwise, you pay full re-prefill on a cold node. This is session affinity for KV cache.

LMCache: offload KV to host memory or storage

GPU memory is scarce and expensive. LMCache moves evicted KV entries to host DRAM or NVMe, creating a tiered cache hierarchy. Reloading from host memory is faster than recomputing from scratch.

The stack is converging

vLLM + cache-aware routing + LMCache + disaggregated prefill. Each layer is a systems problem: scheduling, placement, eviction, transfer.

Walk through the emerging systems stack. Cache-aware routing is sticky sessions for GPUs. LMCache is a write-through cache hierarchy. Disaggregated prefill is compute/storage separation. Each layer maps directly to a distributed systems primitive that the audience already understands. The key insight is that these are not separate projects; they're converging into a coherent systems architecture for inference. vLLM provides the engine, LMCache extends the cache hierarchy, and cache-aware routing makes the scheduler cache-conscious. This is a systems architecture problem, built on the same patterns as CDN edge caching, database buffer pools, and distributed key-value stores.

One more thing: benchmark with real data

// Default benchmark document:
"0 hi hi hi hi hi hi hi..."
// 10,000 reps. 2 unique tokens.
// Token diversity: 0.02%

This overstates compression, distorts cache behavior, and gives misleading throughput numbers.

// Real medical corpus:
"ED NOTE: 47F presenting..."
// 1,329 unique tokens.
// Token diversity: 14.2%

Real trained model weights produce high-entropy KV activations. Compression gains are modest, not magical.

Systems lesson: your benchmark is only as good as its workload model. Use real weights, real documents, and realistic access patterns.
Quick detour on benchmarking methodology. Many KV cache benchmarks use synthetic inputs like "hi hi hi" repeated thousands of times. This creates only 2 unique tokens, produces unrealistically compressible KV activations, and gives misleading results for compression and transfer cost. We built a 30-document corpus with real medical and legal text (300K tokens, generated with real Qwen3-8B-FP8 weights). The difference: 700x more token diversity, and compression ratios that match what you'll see in production. The systems engineering lesson: benchmark with production-representative workloads, or your capacity plans will be wrong.
19 / 20 The systems stack

How to operate this in production

  1. Estimate your knee before deployment. Three numbers: available KV memory, per-session unique context, KV precision. Get within 20%.
  2. Verify with a concurrency sweep. Run C=1 to 2x your expected peak. Find the cliff. Set your limit below it.
  3. Monitor p95/p99, not averages. Alert on the gap between p50 and p99. When it widens sharply, you have crossed the knee.
  1. Separate background from interactive. Background agents can absorb full prefill cost. Reserve cache resources for latency-sensitive workloads.
  2. Use the cheapest lever first. KV quantization (fp16 to fp8) is a config change that buys 75% more headroom. Try it before buying more GPUs.
Operational checklist. These are the concrete things to do on Monday morning. (1) Calculate your expected knee. (2) Run a sweep to verify. (3) Set up tail latency monitoring. (4) Implement workload-based admission control. (5) Try KV quantization before scaling out. Each of these is a systems engineering task, not an ML task. The audience should leave feeling like they have a playbook, not just theory. The monitoring point is critical: most inference dashboards show average TTFT. That will hide the cliff until users are already suffering.
